Polymarket BTC 15-Minute Up/Down Tracker

A live tracker and a backtest pipeline for Polymarket's recurring 15-minute Bitcoin Up/Down markets, built on the official polymarket-client Python package. Source: docs.polymarket.com/dev-tooling/python.

Confirmed: resolution rule & oracle

Each market resolves Up if the BTC price at the end of its 15-minute window is ≥ the price at the start of that window, else Down. The resolution source is Chainlink's BTC/USD Data Stream (data.chain.link/streams/btc-usd) — not Binance spot or any other source. See the note below on what this changes for the pipeline.

Phases 1-4 (tracker + backtest) need zero API keys

Market data, order books, price history, and realtime streams are all served by AsyncPublicClient, which is unauthenticated. You can build and run the entire tracker and backtesting pipeline with no Polymarket account, no wallet, and no secrets.

Phase detail

1

Market discovery

Use search() / list_markets() to find the recurring "Bitcoin Up or Down" 15-minute series and confirm the recurring slug pattern, per-interval condition_id, and Up/Down token_ids.

Since a new market opens every 15 minutes, run a small scheduler that resolves the next market's ids a little before the current one closes, so the pipeline and tracker never miss a rollover.

2

Data pipeline

Async worker subscribes to MarketSpec(token_ids=[...]) for book/price ticks and CryptoPricesSpec(topic="prices.crypto.binance", symbols=["BTCUSDT"]) as a free live proxy for the underlying BTC price.

Normalize rows — timestamp, market_id, token_id, best_bid, best_ask, midpoint, last_trade_price, btc_spot_price — into SQLite/Postgres/Parquet. Backfill closed markets via get_price_history(), and store each market's final resolved outcome straight from Polymarket as the backtest label — this is the actual Chainlink-derived result, so no separate oracle call is needed just to get ground truth.

3

Live tracker

FastAPI + WebSocket app (or a Textual terminal UI) consuming the same realtime subscription. Shows: countdown to resolution, live implied probability (midpoint), order book depth, BTC spot price and its delta from the market's open/strike price, and spread.

4

Backtest engine

Event-driven replay of stored ticks. Pluggable strategy interface: given an order book + BTC spot snapshot, return BUY / SELL / HOLD and size. Simulate fills against the recorded book with slippage.

Metrics: PnL, win rate, Sharpe, max drawdown — plus a parameter-sweep / walk-forward harness for tuning strategies without overfitting.

5

Live execution (optional, later)

Once a strategy is validated in backtest, wire up AsyncSecureClient with place_limit_order / place_market_order, gated behind the credentials in the collapsible table above. Treat this as a separate go/no-go decision, not a default next step.

Open questions to resolve before building

  • Public realtime access: the docs describe CryptoPricesSpec / MarketSpec subscriptions but don't explicitly confirm they work unauthenticated on AsyncPublicClient (only that UserSpec requires the secure client). Verify this empirically early — if it turns out to need auth, Phase 2/3 would need a lightweight secure client even without trading.
  • Rate limits: not documented on this page. Should be measured directly (watch for RateLimitError) once the pipeline is polling/subscribing continuously.

Basis risk: Binance proxy vs. the real Chainlink resolution feed

Binance spot price (via the free CryptoPricesSpec) tracks Chainlink's BTC/USD stream closely almost all the time, but the two can diverge by small amounts, and it's exactly the close-to-even markets where that divergence decides Up vs. Down. Two ways to handle it:

  • Backtesting: unaffected. We label outcomes from Polymarket's own already-resolved result, so backtest accuracy doesn't depend on reproducing Chainlink's number ourselves.
  • Live strategy signal, near the boundary: a strategy trading close calls in real time is exposed to this basis risk. Closing it exactly requires Chainlink Data Streams, which is a commercial product needing its own API credentials (contact Chainlink to get a Client ID/Secret) — separate from anything Polymarket issues. Treat as a Phase 3/4 nice-to-have, not a blocker: start with the free Binance proxy and only add Chainlink credentials if backtests show edge-case markets are where the strategy's edge lives.
